    PROGRAMMER'S VIEW ON PORTABILITY AND POWER
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

        TI/2 is written entirely in C/C++.  The authors choose to use
    absolutely no Assembly language or detection of output hardware etc.
    Since Assembly was not designed as a portable language and output
    hardware, including video cards, varies from machine to machine
    programmers who rely on detecting it tend to bind their code to a
    specific operating system.  There choices have allowed TI/2 to have the
    power that it does.  From the base up TI/2 has been designed as a
    portable development library that not only spans multiple compilers, but
    platforms also.  The design of the routines that make up the library are
    so flexible that the interface could even be implemented over a modem
    using a terminal emulation such as ANSI or Avatar or on a remote
    terminal.  The source code is written with defines and function wrappers
    to allow for the greatest amount of portability by reducing system and
    platform dependency.  This portability will allow developers who use
    TI/2 to develop multi-platform products with greater ease by reducing
    the amount of source code porting required on their part.

        TI/2 is superior to other text interface kits because of many
    reasons. Since the source code is available, developers will not have to
    give excuses to their customers because the interface library they use
    does not support their target platform.  Many applications in the past
    have faded away because they relied on libraries where the source code
    was unavailable or too system dependent.  The availability of source
    code doesn't mean that it is understandable source code.  The TI/2
    source code if fully commented and laid out in a logical manner to allow
    developers to become familiar with it quickly and adapt it to suit their
    needs.

    DEFINITION OF A WINDOW
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

        A window is a region of the screen that is separate and independent
    from the other regions of the screen.  Windows are a vital part of an
    application and help to provide a profession appearance.  Windows are
    used to construct many things including pull-down menus.  The use of
    windows is found in almost all profession programs.  A great example of
    the use of windows to create an integrated environment is the Borland
    compiler's IDE.

    PROPERTIES OF TI/2 WINDOWS
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

        - More than one window can be displayed on the screen
          simultaneously.

        - Windows are allowed to overlap without disturbing the contents of
          the windows underneath.

        - A Window that is hidden underneath other windows can be brought to
          the top of the screen.

        - Windows can be disposed of at any time.  When erased the windows
          above and below are updated and restored correctly.

        - Windows can be written to or redraw at any time.

    GLOBAL VARIABLES
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

    the variables that are declared in programs that use TI/2 have almost no

    DESKTOP
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

        When the TI/2 library is initialized it creates a desktop window.
    This is necessary because of the way TI/2 handles window management.
    The desktop window covers the entire screen and can be used be used by
    you in your program.  The desktop window is named desktop so you can use
    it like any other window.  For instance if you want to print something
    on the desktop just add the command desktop.wputs("Hello world!"); to
    your program.  Some functions should not be used with the desktop
    window.  For instance those that hide, unhide, make it see-through, etc.
    There is need for you to call desktop.destroy at the end of your program
    either.  This is done automatically when TI/2 is deinitialized.

    OVERLAPPING WINDOWS AND OUTPUT
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

        With TI/2, if a window overlaps another window, this overlap will be
    displayed on the physical screen.  TI/2 allows you to direct output to
    any window at any time.  When output is written to a window, the windows
    contents are updated in memory, but only the visible portions of the
    window will be updated on the screen.

    MANDATORY FUNCTIONS AND HEADER FILES
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

        header files
        ÄÄÄÄÄÄÄÄÄÄÄÄ

            The only mandatory functions are win_init() and win_deinit()
        which are explained below.  The ti2.h header file must be included
        in any module that will call functions that are part of the TI/2
        library.  This means that in any C++ file you must have the
        following line with the other #include statements.

            #include "ti2.h"

        win_init()
        ÄÄÄÄÄÄÄÄÄÄ

            This function must be called before any function defined in the
        TI/2 library.  Failure to do so will cause all TI/2 functions not to
        work.  The purpose of this function is to setup global variables and
        allocate any memory that TI/2 might need.  After all the
        initialization of variables and memory the desktop window is
        created.

        win_deinit()
        ÄÄÄÄÄÄÄÄÄÄÄÄ

            This function needs to be called right before your program
        exits. The purpose of this function is to perform cleanup, memory
        deallocation, and destroy the desktop window. If this function is
        not called it could cause a "memory leak" in your program that the
        operating system might not correct.

        Example
        ÄÄÄÄÄÄÄ

        #include "ti2.h"

        void main(void)
            {
              /* Initialize TI/2 */
              win_init();

              /* ... Program code goes here ... */


              /* Deinitialize TI/2 */
              win_deinit();
            }

    win_redraw_all
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

    Function      Redraw all windows on screen.

    Syntax        void win_redraw_all(void);

    Remarks       win_redraw_all redraws all the windows on the screen
                  starting with the desktop and windowing up to the top
                  window.  This function may needed to be called if the
                  screen gets messed up for some unknown reason and needs a
                  refresh.  This function is also called when the text mode
                  is changed.

    Return value  None.

    See Also      win_text_mode

    win_set_default_colors
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

    Syntax        void win_set_default_colors(uchar f_fcolor,
                                              uchar f_bcolor,

                                              uchar t_bcolor,

    Syntax        void win_text_mode(int newmode);

    Remarks       win_text_mode changes the text mode, adjusts the desktop

        For those who are unfamiliar with the way classes work here is a
    brief description of how to declare a window.  Windows are like any
    other variables.  To create a window variable just declare it like this:

        win_c my_window;      // This is a window variable

        After declaring a window variable you will need to call the create
    function to open the window.  Then you can call any of the other
    functions that are part of the window class.  When you are finished with
    the you can remove is from the screen by calling the destroy function.
    If you are new to C++ than here is an example of calling some window
    functions.

            my_window.clr_eol();
            my_window.wprintf("Hello World! I'm %d years old\n", 16);

        The following functions are part of the window class.

        Windows are not complete without controlled ways to input data from
    the user.  TI/2 has an extensive set of functions to input different
    types of data.  These functions are part of the win_c class and are
    called just like the other functions.  All input functions use the input
    colors specified for the window by the input_init function.

    ** These functions where mainly used for the old data entry style, but
       have been replaced by a more powerful form system.  These functions
       may be removed in the first official release to cut down on the size
       of the library.

    Remarks       input_frmt_string inputs a string from the user only
                  allowing them to type alpha and numeric characters in
                  certain places.  x and y are the coordinates inside the

    MENU CLASS FUNCTIONS
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

        Pull down menus are a necessity in any program that deals with many
    options and commands.  TI/2 has was designed to make menu creation as
    easy as possible.  When creating a menu there are two types of IDs.
    There are menu command and submenu IDs.  When designing a pull down menu
    it is recommended that you use #define statements to keep the IDs from
    becoming confusing.  Here is an example of the #define statement section
    for a menu.

    /* Submenus */
    #define SM_FILE   1
    #define SM_EDIT   2

    /* Menucmds on File menu */

    Notice the style of the define section.  All submenus are preceded by a
    SM_ and all menu commands are preceded by a MC_.  No two submenus can
    have the same ID number.  No two menu commands can have the same ID
    number.  No menu command or submenu ID can be less than one.  It is
    recommended that you start numbering at one and go up sequentially.

    MOUSE FUNCTIONS

      Mouse support adds simplicity to the form and menu classes.  The mouse
    support provided is non-interrupt driven and available on both the DOS
    and OS/2 systems.  Detection of the mouse is done automatically by the
    win_init function.  The mouse is reset when the win_deinit function is
    called.  Calls to mouse functions without the presence of a mouse will
    not affect the program since TI/2 keeps track of wether the mouse is
    there or not.  There are global variables which the mouse system
    maintains.  These variables are:

    mouse_exist   - True if mouse exists
    mouse_left    - Status of left button
    mouse_center  - Status of center button
    mouse_right   - Status of right button
    mouse_row     - Current row of mouse cursor
    mouse_col     - Current column of mouse cursor

    There are defined values for the buttons.  These defines are:

    0               - Button not down or clicked
    BUTTON_CLICKED  - Button clicked
    BUTTON_DOWN     - Button is currently down

    TIME SLICE FUNCTIONS
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

      Since multitasking operating systems are a reality programs should be
    designed with CPU usage in mind.  Whenever a program is in an idle loop
    the time slice should be called.  This allows the operating system to
    give the CPU time to another program.  TI/2 supports OS/2, DESQview,
    Windows, and Windows/386 time slicing.  Below are the functions used for
    time slicing.

    init_timeslice
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

    Function      Detects the operating system and time slicing type to use.

    Syntax        void init_timeslice(void);

    Remarks       init_timeslice is automatically called by win_init and
                  detects the multitasker that TI/2 is running under.

    Return value  none.

    timeslice
    ÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄÄ

    Function      Releases time to the operating system.

    Syntax        void timeslice(void);

    Remarks       timeslice gives up time to other programs running under
                  the multitasker.  This function should be called when
                  the program is idlely waiting for input.

    Return value  none.
